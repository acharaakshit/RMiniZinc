Introduction

MiniZinc is a free and open-source constraint modeling language. Constraint satisfaction and discrete optimization problems can be formulated in a high-level modeling language. Models are compiled into an intermediate representation that is understood by a wide range of solvers. MiniZinc itself provides several solvers, for instance GeCode. The existing packages in R are not powerful enough to solve even mid-sized problems in combinatorial optimization

There are implementations of an Interface to MiniZinc in Python like MiniZinc Python and pymzn and JMiniZinc for Java but such an interface does not exist for R.

This package provides an implementation of a very simple and easy to use interface for R that will help R users to solve optimization problems that can’t be solved with R currently.

It’s important to understand R6 classes before getting into the details. If you are not comfortable with R6, please go through this tutorial.

It would be nice to go through this manual to understand more about MiniZinc. This is mainly for those who are interested in contributing to the package.

Please make sure to install MiniZinc on your operating system and also ensure that minizinc is accessible from command line by typing ‘minizinc’ on the command line without the single quotes. The installation instructions can be found at instructions

Variables

There are two types of variables in MiniZinc – decision variables and parameters.

The data types of variables can be single types i.e integers (int), floating point numbers (float), Booleans (bool) and strings (string) and collections i.e sets, enums and arrays (upto 6 dimensional arrays).

Parameter is used to specify a parameter in a given problem and they are assigned a fixed value or expression.

Decision variables are the unknowns that Minizinc model is finding solutions for. We do not need to give them a value, but instead we give them a domain of possible values. Sometimes expressions involving other variables and parameters are also assigned to decision variables. Decision variables need to satisfy a set of constraints which form the core of the problem.

To create a variable declaration one needs to understand the elements of R6 classes that will be used to create the variables.

The type information of the declaration is stored in an object of class Type which holds information like base type, number of dimensions, etc. The type inst is stored in an object of TypeInst class which contains the type information as well as the index or domain in case the variable is an array or decision variable with domain respectively. The value assigned to the variable declaration is stored in an object of sub classes of super class Expression that can be Int, Float, Array, or other Expression sub classes that result in a value of corresponding type.

Easy to use declaration functions have been created for the users to declare variables and parameters of different data types. Examples of how to declare variables is shown below.

Here are examples of how we can create parameters and decision variables.

library(rminizinc)

# create the variable and parameter declarations
item1 = VarDeclItem$new(decl = IntDecl(name = "n", kind = "par"))

par2_val = BinOp$new(lhs_expression = Int$new(1), binop = "..", rhs_expression = item1$e()$id())
item2 = VarDeclItem$new(decl = IntSetDecl(name = "OBJ", kind = "par", value = par2_val))

item3 = VarDeclItem$new(decl = IntDecl(name = "capacity", kind = "par"))

item4 = VarDeclItem$new(decl = IntArrDecl(name = "profit", kind = "par", ndim = 1, ind = list(item2$e()$id())))

item5 = VarDeclItem$new(decl = IntArrDecl(name = "size", kind = "par", ndim = 1, ind = list(item2$e()$id())))

item6 = VarDeclItem$new(decl = IntArrDecl(name = "x", kind = "var", ndim = 1, ind = list(item2$e()$id())))

Constraints

Constraints are defined on the decision variables to restrict the range of values that they can take. They can also be thought of as the rules of a problem.

Constraints can be created using different R6 sub classes of the super class Expression. In this example Generator, Binop, Comprehension and Call classes have been used. These classes take in the elements required to create an expression that will be used as a constraint. More information can be found using ?<class Name>

Create constraints:

# declare parameter for iterator
parIter = IntDecl(name = "i", kind = "par")


gen_forall = Generator$new(IN = item2$e()$id(), decls = list(parIter))
bop1 = BinOp$new(lhs_expression = ArrayAccess$new(v = item6$e()$id(), 
                args= list(gen_forall$decl(1))),
                binop = ">=", rhs_expression = Int$new(0))

Comp1 = Comprehension$new(generators = list(gen_forall), e = bop1, set = FALSE)
cl1 = Call$new(fnName = "forall", args = list(Comp1))
item7 = ConstraintItem$new(e = cl1)

gen_sum = Generator$new(IN = item2$e()$id(), decls = list(parIter))

bop2 = BinOp$new(lhs_expression = ArrayAccess$new(v = item5$e()$id(), args = list(gen_sum$decl(1))),                  binop = "*",  rhs_expression = ArrayAccess$new(v = item6$e()$id() , 
                args = list(gen_sum$decl(1))))
Comp2 = Comprehension$new(generators = list(gen_sum), e = bop2, set = FALSE)
cl2 = Call$new(fnName = "sum", args = list(Comp2))
bop3 = BinOp$new(lhs_expression = cl2, binop = "<=", rhs_expression = item3$e()$id())
item8 = ConstraintItem$new(e = bop3)

Solve Type

The constraint programming problem can be of three types, namely: Satisfaction , Minimization and Maximization. Satisfaction problems produce all the solutions that satisfy the constraints whereas minimization and maximization problems produce the solution which minimizes and maximizes the given arithmetic expression.

An example is shown below:

bop4 = BinOp$new(lhs_expression = ArrayAccess$new(v = item4$e()$id(), args = list(gen_sum$decl(1))),
                      binop = "*", rhs_expression = ArrayAccess$new(v = item6$e()$id(), 
                      args = list(gen_sum$decl(1))))

Comp3 = Comprehension$new(generators = list(gen_sum), e = bop4, set = FALSE)

cl3 = Call$new(fnName = "sum", args = list(Comp3))

item9 = SolveItem$new(solve_type = "maximize", e = cl3)

Create the model

Currently only three types of items can be created in a MiniZinc model namely: Variable Declarations, Constraints and Solve Items. The model is created by combining these items.

Combine all the items to create a MiniZinc model

items  = c(item1, item2, item3, item4, item5, item6, item7, item8, item9)
mod = Model$new(items = items)
modString = mod$mzn_string()
cat(modString)
## int: n;
## set of int: OBJ = 1 .. n;
## int: capacity;
## array[OBJ] of int: profit;
## array[OBJ] of int: size;
## array[OBJ] of var int: x;
## constraint forall([x[i] >= 0 | i in OBJ ]);
## constraint sum([size[i] * x[i] | i in OBJ ]) <= capacity;
## solve  maximize sum([profit[i] * x[i] | i in OBJ ]);

Solve the model using the Rcpp functions that are using the MiniZinc.

sol = rminizinc:::mzn_eval(modelString = modString, solver = "org.gecode.gecode",

Parse Model, Set Missing Parameters and Solve

A parser function mzn_parse has also been implemented which can be used to detect possible syntax errors and get the smallest of details before the MiniZinc model is evaluated. This function can either take the path of a model file path or it can take a string model and returns a list with different elements like: MODEL_STRING which is the string representation of the model and VARIABLES which are the variable declaration details, CONSTRAINTS which are the constraint details of a model, SOLVE_TYPE which contains the information about the solve type of the problem, FUNCTION_ITEMS contains the function information, ASSIGNMENTS contains the assignment information, INCLUDES contains information about the included mzn files and OUTPUT_ITEM contains the information about the output formatting. Each component (Variable Declarations, Constraints, Solve Items, Assignment Items, Include Items, Functions and Assignments) in a MiniZinc model is considered as an item. The ITEM_NO for each element is also returned.

The Information about the naming of the elements returned is described in the following figures:

Model Items

Name: VARIABLES
Components: DECL<n> is the list of declaration with components DETAILS and ITEM_NO
Description: DETAILS are the details of a variable declaration namely: NAME, TYPE, KIND, DOMAIN and VALUE and ITEM_NO is the item number of the variable declaration in the model.

Name: CONSTRAINTS
Components: CONSTRAINT<n> is the list of constraints with components: DETAILS and ITEM_NO.
Description: DETAILS are the details about the expressions.

Name: FUNCTION_ITEMS
Components: FUNCTION<n> is the list of functions with components: DETAILS and ITEM_NO.
Description: DETAILS are the details that can EXPRESSION, ANNOTATION or PARAMETER_DECLARATIONS.

Name: SOLVE_TYPE
Components: DETAILS, OBJECTIVE and ITEM_NO
Description: DETAILS are the details that can EXPRESSION or ANNOTATION, OBJECTIVE can be “satisfy”, “minimize” or “maximize”.

Name: ASSIGNMENTS
Components: ASSIGNMENT<n> have the sub components: DETAILS and ITEM_NO.
Description: DETAILS contain the value expression details.

Name: INCLUDES
Components: INCLUDED_MZN and ITEM_NO Description: INCLUDED_MZN is the name of the included mzn file.

Name: OUTPUT_ITEM
Components: DETAILS and ITEM_NO.
Description: DETAILS are the details about the output formatting expression.

Basic Expressions

These are the expressions that lie at the lowest level when parsing any expression.

Parent_Expressions

These are the expressions that are above the basic types in hierarchy and can have sub expressions that can be either basic expressions or parent expressions.

Properties of Parent Expressions

Each parent has different properties that define it’s structure. The properties of each of the parent are described below:

Name: COMPREHENSION
Components: GENERATORS and EXPRESSION
Description: GENERATORS is a list of expressions and each generator contains DECLARATIONS (list of variable declarations), IN_EXPRESSION and WHERE_EXPRESSION (Expressions). and EXPRESSION is the expression in the comprehension.

Name: BINARY_OPERATION
Components: LHS, BINARY_OPERATOR and RHS
Description: LHS and RHS are both expressions (basic or parent) and BINARY_OPERATOR is a binary operator in MiniZinc.

Name: UNARY_OPERATION
Components: UNARY_OPERATOR and ARGUMENTS
Description: UNARY_OPERATOR are the unary operators in MiniZinc and ARGUMENTS contain a list of expressions.

Name: VARIABLE_DECLARATION
Components: NAME, KIND, TYPE, DOMAIN and VALUE
Description: NAME, KIND and TYPE are strings representing the name, kind and type of the variable and DOMAIN and VALUE are expressions representing domain and value of the variable declaration.

Name: ARRAY_ACCESS
Components: NAME and ARGUMENTS
Description: NAME is the name of the variable and ARGUMENTS is the list of expressions that represent the indices.

Name: IF_THEN_ELSE
Components: IF, THEN and ELSE
Description: IF is the list of if expressions, THEN is the list of then expressions and ELSE is the else expression.

Name: LET
Components: LET_EXPRESSION and IN_EXPRESSION
Description: LET_EXPRESSION and IN_EXPRESSION are lists of expressions.

Name: CALL
Components: NAME and ARGUMENTS
Description: NAME is the name of the function call and ARGUMENTS is the list of expressions to that function call.

Name: SET
Components: MIN, MAX (set ranges) or ELEMENT<n> where n is the serial number of the element.
Description: MIN , MAX and ELEMENT<n> are all expression types (mostly basic expressions).

Name: ARRAY
Components: ELEMENT<n> where n is the serial number of the element.
Description: ELEMENT<n> is an expression.

The function set_params() can be used to set the missing parameters of the model. Any number of missing parameters can be set. The values for n dimensional arrays or sets can be provided as vectors like c(1,2,..,n). The indices of the arrays will be automatically retrieved and the values of the arrays will be stored as arraynd() functions. Set ranges will be supported soon. The missing parameters can be obtained using getMissingPars().

The function mzn_eval() is used to evaluate a MiniZinc model and returns the solution string and a list of solutions if they were parsed without any error by the function sol_parse() otherwise it returns the solution string and an appropriate error. The optimal solution if found can be accessed using OBJ$SOLUTIONS$OPTIMAL_SOLUTION and the best solution can be accessed using OBJ$SOLUTIONS$BEST_SOLUTION. More details about the functions can be obtained using ?mzn_eval and ?sol_parse.

A sample job shop problem has been solved below:

# mzn file path
mzn_path = paste0(dirname(getwd()), "/inst/extdata/mzn_examples/jobshop/jobshop_0.mzn")

# parse the model
parseObj=rminizinc:::mzn_parse(mznpath = mzn_path)

# get the modelString
modString = parseObj$MODEL_STRING

# dzn file path
dzn_path = paste0(dirname(getwd()), "/inst/extdata/mzn_examples/jobshop/jobshop.dzn")

# R List object containing the solutions

The data can also be provided as a list of values of parameters where the names of the list must be the names(whole or part) of the parameters whose values are missing. The names of the parameters whose values are missing can be obtained using the function getMissingPars. Alternatively, if the user wishes to provide a data file, it can be provided to the function mzn_eval().

# file path
mzn_path = paste0(dirname(getwd()), "/inst/extdata/mzn_examples/knapsack/knapsack_0.mzn")

NOTE: Please don’t include output formatting in the mzn files or the solutions might not be parsed properly.

The named list returned by mzn_parse() can be used to initialize all the R6 classes required to create the model that was parsed. This will help the user to use a lot of functions to understand, manipulate and use the model using the R6 objects. getRModel() is used to initialize all the relevant objects using named list returned by mzn_parse(). The support is only available for basic models currently but the scope will be increased soon.

# mzn file path
mzn_path = paste0(dirname(getwd()), "/inst/extdata/mzn_examples/production_planning/prod_plan_0.mzn")

# parse the model
parseObj=rminizinc:::mzn_parse(mznpath = mzn_path)

modString = getRModel(parseObj)$mzn_string()
