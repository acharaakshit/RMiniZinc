MiniZinc Interface for R

Akshit Achara, Lars Kotthoff

2020-08-22

Introduction

MiniZinc is a free and open-source constraint modeling language. Constraint satisfaction and discrete optimization problems can be formulated in a high-level modeling language. Models are compiled into an intermediate representation that is understood by a wide range of solvers. MiniZinc itself provides several solvers, for instance GeCode. The existing packages in R are not powerful enough to solve even mid-sized problems in combinatorial optimization.

There are implementations of an Interface to MiniZinc in Python like MiniZinc Python and pymzn and JMiniZinc for Java but such an interface does not exist for R.

This package provides an implementation of a very simple and easy to use interface for R that will help R users to solve optimization problems that can’t be solved with R currently.

It’s important to understand R6 classes before getting into the details. If you are not comfortable with R6, please go through this tutorial.

It would be nice to go through this manual to understand more about MiniZinc. This is mainly for those who are interested in contributing to the package.

Please make sure to install MiniZinc on your operating system and also ensure that minizinc is accessible from command line by typing ‘minizinc’ on the command line without the single quotes. The installation instructions can be found here.

Variables

There are two types of variables in MiniZinc namely, decision variables and parameters.

The data types of variables can be single types i.e integers (int), floating point numbers (float), Booleans (bool) and strings (string) and collections i.e sets, enums and arrays (upto 6 dimensional arrays).

Parameter is used to specify a parameter in a given problem and they are assigned a fixed value or expression.

Decision variables are the unknowns that Minizinc model is finding solutions for. We do not need to give them a value, but instead we give them a domain of possible values. Sometimes expressions involving other variables and parameters are also assigned to decision variables. Decision variables need to satisfy a set of constraints which form the core of the problem.

To create a variable declaration one needs to understand the elements of R6 classes that will be used to create the variables.

Easy to use declaration functions have been created for the users to declare variables and parameters of different data types. Examples of how to declare variables is shown below.

library(rminizinc)

# create the variable and parameter declarations
item1 = VarDeclItem$new(decl = IntDecl(name = "n", kind = "par"))

par2_val = BinOp$new(lhs = Int$new(1), binop = "..", rhs = item1$id())
item2 = VarDeclItem$new(decl = IntSetDecl(name = "OBJ", kind = "par", value = par2_val))

item3 = VarDeclItem$new(decl = IntDecl(name = "capacity", kind = "par"))

item4 = VarDeclItem$new(decl = IntArrDecl(name = "profit", kind = "par", ndim = 1, 
                                          ind = list(item2$id())))

item5 = VarDeclItem$new(decl = IntArrDecl(name = "size", kind = "par", ndim = 1, ind =                                                            list(item2$id())))

item6 = VarDeclItem$new(decl = IntArrDecl(name = "x", kind = "var", ndim = 1, ind = list(item2$id())))

Constraints

Constraints are defined on the decision variables to restrict the range of values that they can take. They can also be thought of as the rules of a problem.

Constraints can be created using different R6 sub classes of the super class Expression. In this example Generator, Binop, Comprehension and Call classes have been used. These classes take in the elements required to create an expression that will be used as a constraint. More information can be found using ?<class Name>

Create constraints:

# declare parameter for iterator
parIter = IntDecl(name = "i", kind = "par")


gen_forall = Generator$new(IN = item2$id(), decls = list(parIter))
bop1 = BinOp$new(lhs = ArrayAccess$new(v = item6$id(),  args= list(gen_forall$decl(1))),
                                                             binop = ">=", rhs = Int$new(0))

Comp1 = Comprehension$new(generators = list(gen_forall), body = bop1, set = FALSE)
cl1 = Call$new(fnName = "forall", args = list(Comp1))
item7 = ConstraintItem$new(e = cl1)

gen_sum = Generator$new(IN = item2$id(), decls = list(parIter))

bop2 = BinOp$new(lhs = ArrayAccess$new(v = item5$id(), args = list(gen_sum$decl(1))),                  
                 binop = "*",  rhs = ArrayAccess$new(v = item6$id() , 
                 args = list(gen_sum$decl(1))))

Comp2 = Comprehension$new(generators = list(gen_sum), body = bop2, set = FALSE)
cl2 = Call$new(fnName = "sum", args = list(Comp2))
bop3 = BinOp$new(lhs = cl2, binop = "<=", rhs = item3$id())
item8 = ConstraintItem$new(e = bop3)

Solve Type

The constraint programming problem can be of three types, namely: Satisfaction , Minimization and Maximization. Satisfaction problems produce all the solutions that satisfy the constraints whereas minimization and maximization problems produce the solution which minimizes and maximizes the given expression.

An example is shown below:

bop4 = BinOp$new(lhs = ArrayAccess$new(v = item4$id(), args = list(gen_sum$decl(1))),
                      binop = "*", rhs = ArrayAccess$new(v = item6$id(), 
                      args = list(gen_sum$decl(1))))

Comp3 = Comprehension$new(generators = list(gen_sum), body = bop4, set = FALSE)

cl3 = Call$new(fnName = "sum", args = list(Comp3))

item9 = SolveItem$new(solve_type = "maximize", e = cl3)

Parse Model, Set Missing Parameters and Solve

Parse a MiniZinc Model

A parser function mzn_parse has also been implemented which can be used to detect possible syntax errors and get the smallest of details before the MiniZinc model is evaluated. This function can either take the path of a model file path or it can take a string model and returns a list with different elements like: MODEL_STRING which is the string representation of the model and VARIABLES which are the variable declaration details, CONSTRAINTS which are the constraint details of a model, SOLVE_TYPE which contains the information about the solve type of the problem, FUNCTION_ITEMS contains the function information, ASSIGNMENTS contains the assignment information, INCLUDES contains information about the included mzn files and OUTPUT_ITEM contains the information about the output formatting. Each component (Variable Declarations, Constraints, Solve Items, Assignment Items, Include Items, Functions and Assignments) in a MiniZinc model is considered as an item. The ITEM_NO for each element is also returned. If your model includes an mzn file, you can provide custom paths of the folders where the included files are present. The default mzn files (in the std folder) provided along with libminizinc will be included by default.

The Information about the naming of the elements returned is described in the following figures:

Model Items

Name: VARIABLES
Components: DECL<n> is the list of declaration with components DETAILS and ITEM_NO
Description: DETAILS are the details of a variable declaration namely: NAME, TYPE_INST and VALUE and ITEM_NO is the item number of the variable declaration in the model.

Name: CONSTRAINTS
Components: CONSTRAINT<n> is the list of constraints with components: DETAILS and ITEM_NO.
Description: DETAILS are the details about the expressions.

Name: FUNCTION_ITEMS
Components: FUNCTION<n> is the list of functions with components: DETAILS and ITEM_NO.
Description: DETAILS are the details that can EXPRESSION, ANNOTATION, TYPE_INST PARAMETER_DECLARATIONS.

Name: SOLVE_TYPE
Components: DETAILS, OBJECTIVE and ITEM_NO
Description: DETAILS are the details that can EXPRESSION or ANNOTATION, OBJECTIVE can be “satisfy”, “minimize” or “maximize”.

Name: ASSIGNMENTS
Components: ASSIGNMENT<n> have the sub components: DETAILS and ITEM_NO.
Description: DETAILS contain the value expression details.

Name: INCLUDES
Components: INCLUDED_MZN and ITEM_NO Description: INCLUDED_MZN is the name of the included mzn file.

Name: OUTPUT_ITEM
Components: DETAILS and ITEM_NO.
Description: DETAILS are the details about the output formatting expression.

Basic Expressions

These are the expressions that lie at the lowest level when parsing any expression.

Parent Expressions

These are the expressions that are above the basic types in hierarchy and can have sub expressions that can be either basic expressions or parent expressions.

Properties of Parent Expressions

Each parent has different properties that define it’s structure. The properties of each of the parent are described below:

Name: COMPREHENSION
Components: GENERATORS and EXPRESSION
Description: GENERATORS is a list of expressions and each generator contains DECLARATIONS (list of variable declarations), IN_EXPRESSION and WHERE_EXPRESSION (Expressions). and EXPRESSION is the expression in the comprehension.

Name: BINARY_OPERATION
Components: LHS, BINARY_OPERATOR and RHS
Description: LHS and RHS are both expressions (basic or parent) and BINARY_OPERATOR is a binary operator in MiniZinc.

Name: UNARY_OPERATION
Components: UNARY_OPERATOR and ARGUMENTS
Description: UNARY_OPERATOR are the unary operators in MiniZinc and ARGUMENTS contain a list of expressions.

Name: VARIABLE_DECLARATION
Components: NAME, TYPE_INST (KIND, TYPE, DOMAIN, INDEX) and VALUE
Description: NAME, KIND and TYPE are strings representing the name, kind and type of the variable and DOMAIN and VALUE are expressions representing domain and value of the variable declaration. TYPE_INST contains the KIND, TYPE, DOMAIN and INDEX.

Name: ARRAY_ACCESS
Components: NAME and ARGUMENTS
Description: NAME is the name of the variable and ARGUMENTS is the list of expressions that represent the indices.

Name: IF_THEN_ELSE
Components: IF, THEN and ELSE
Description: IF is the list of if expressions, THEN is the list of then expressions and ELSE is the else expression.

Name: LET
Components: LET_EXPRESSION and IN_EXPRESSION
Description: LET_EXPRESSION and IN_EXPRESSION are lists of expressions.

Name: CALL
Components: NAME and ARGUMENTS
Description: NAME is the name of the function call and ARGUMENTS is the list of expressions to that function call.

Name: SET
Components: MIN, MAX (set ranges) or ELEMENT<n> where n is the serial number of the element.
Description: MIN , MAX and ELEMENT<n> are all expression types (mostly basic expressions).

Name: ARRAY
Components: ELEMENTS and DIM_SIZE
Description: ELEMENTS have sub fields to type ELEMENT<n> where n is the serial number of the element and DIM_SIZE has sub fields DIM<n> which have the MIN and MAX index value of each dimension.

Find the Missing Parameters

The missing parameters can be obtained using get_missing_pars()

missingPars = get_missing_pars(modelString = parseObj$MODEL_STRING)
print(missingPars)
## [1] "n"  "m"  "d"  "mc"

Set Values of Missing Parameters

The function set_params() can be used to set the missing parameters of the model. Any number of missing parameters from 1 to the number of missing parameters can be set. The values for n dimensional arrays or sets can be provided as vectors like c(1, 2, .., n). The indices of the arrays will be automatically retrieved and the values of the arrays will be stored as arraynd() functions. The values should be provided as a named list where the names of each value is the name of the paramaeter. The string representation of the updated model will be returned on call. In case an mzn file path is provided it can also be updated after setting the values by setting modifyMZN to TRUE.

pVals = list(3, 4, c(3, 3, 4, 4, 4, 3, 2, 2, 3, 3, 3, 4), 
             c(1, 2, 3, 4, 1, 3, 2, 4, 4, 2, 1, 3))
names(pVals) = missingPars
modString = set_params(modData = pVals, mznpath = mzn_path)

Solve the Model

The function mzn_eval() is used to evaluate a MiniZinc model and returns the solution string and a list of solutions if they were parsed without any error by the function sol_parse() otherwise it returns the solution string and an appropriate error. The parsed solutions are a named list where elements are of type OBJ$SOLUTIONS$SOLUTION<n>$<VARIABLE_NAME>. The optimal solution if found can be accessed using OBJ$SOLUTIONS$OPTIMAL_SOLUTION and the best solution can be accessed using OBJ$SOLUTIONS$BEST_SOLUTION. More details about the functions can be obtained using ?mzn_eval and ?sol_parse.

The solver that should be used to solve the model needs to be specified by the user (default is Gecode) and the libpath (path of the MiniZinc executable) should also be provided. If the user wishes to provide a data file, it’s path can be provided to the argument dznpath. A time limit (in ms) can also be provided to the argument time_limit. (default is 300000 ms)

A sample job shop problem has been solved below:

# R List object containing the solutions

                     libpath = "/snap/minizinc/current/share/minizinc")

Modify Constraints

There are various getter and setter functions for the expression classes that can be used to modify existing constraints. For example:

vItem = VarDeclItem$new(mzn_str = "set of int: a = {1, 2, 3, 4};") 
cItem = ConstraintItem$new(mzn_str = "constraint sum(a) < 10;")
sprintf("The current constraint is: %s", cItem$c_str())
cItem$setExp(BinOp$new(lhs = Call$new(fnName = "max", args = list(vItem$getDecl()$id())),
                       binop = "<", rhs = Int$new(10)))
sprintf("The modified constraint is: %s", cItem$c_str())
## [1] "The current constraint is: constraint (sum(a) < 10);\n"
## [1] "The modified constraint is: constraint (max(a) < 10);\n"
